# Contributing to TransitOps

## Before you start

1. Create a branch from the current `main` branch.
2. Set up the project using the instructions in [README.md](README.md).
3. Keep each change focused on one feature or fix.

## Development expectations

- Keep secrets in `.env`; never commit credentials, local databases, virtual environments, or generated bytecode.
- Add migrations for model changes with `python manage.py makemigrations <app_name>`.
- Do not edit an applied migration unless the project is still in a fresh-development state and the team agrees.
- Keep templates namespaced under their app directory and use named URLs.
- Preserve role-based access checks and finite-state workflow guards when modifying operational features.

## Verification

Run the checks relevant to your change before opening a pull request:

```bash
python manage.py check
python manage.py test
npm run build
```

For model or migration work, also verify a fresh database can apply migrations:

```bash
python manage.py migrate
```

## Pull requests

Include a concise description, affected apps, migration notes, and test evidence. Call out any environment-variable, database, or external-service changes explicitly.

Avoid committing unrelated files such as duplicate exports, PDFs, editor settings, or local SQLite databases.
